# Brainstorm Context: Fix ArgoCD "app path does not exist" Error

## Problem Statement

Every new service created via `init.sh` produces an ArgoCD error: "Failed to load target state: failed to generate manifest for source 1 of 1: rpc error: code = Unknown desc = k8s: app path does not exist". The root cause is that `.gitignore` (copied from `init/gitignore`) ignores `k8s/*.yaml` and `argocd/*.yaml`, so the rendered manifests are never committed to the app's git repo. ArgoCD's Application resource points to `path: k8s`, but that directory is empty or absent in the remote repo.

## Key Concepts

- **ArgoCD Application**: Points to `path: k8s` in the app's git repo. ArgoCD reads manifests from that path to sync state to the cluster.
- **`.gitignore` template**: `init/gitignore` is copied into every scaffolded app. It ignores `k8s/*.yaml` and `argocd/*.yaml` — exactly the files ArgoCD needs.
- **`init.sh` rendering**: Lines 234-256 render k8s and argocd templates into the app directory. Lines 282-287 do `git init && git add . && git commit` — but gitignored files are excluded from `git add .`.
- **`build.sh` rendering**: Re-generates k8s/argocd manifests at build time (updating image tags). Also writes to the same gitignored paths.

## Approaches Considered

### Approach A: Remove gitignore rules for k8s/argocd YAML files
- **How it works**: Delete `k8s/*.yaml` and `argocd/*.yaml` from `init/gitignore`. These files become tracked by git in every scaffolded app.
- **Tradeoffs**: Simplest fix. Files ArgoCD needs are always committed. But k8s/argocd manifests are now tracked alongside source code, which may feel noisy.
- **Risks**: Low. The files are already rendered by init.sh and are necessary for ArgoCD.
- **Best for**: When the intent is that k8s/argocd manifests live in the app repo and are committed alongside code.

### Approach B: Force-add in init.sh despite gitignore
- **How it works**: Change `git add .` to `git add . && git add -f k8s/*.yaml argocd/*.yaml` in init.sh. Keeps the gitignore for general hygiene but ensures manifests are committed.
- **Tradeoffs**: More surgical. Preserves gitignore intent. But every future `git add` by the developer also needs `-f`, and build.sh outputs remain untracked unless the developer force-adds manually.
- **Risks**: Medium. Developers may not realize they need to force-add after build.sh runs. The gitignore creates a confusing contradiction.
- **Best for**: When you want to keep k8s/argocd out of normal git tracking but still committed at init time.

### Approach C: Remove gitignore rules AND ensure build.sh commits after rendering
- **How it works**: Same as Approach A, plus add `git add k8s/ argocd/ && git commit` to build.sh after template rendering.
- **Tradeoffs**: Most complete. Ensures manifests are always committed, both at init and at build time. Adds a commit to every build, which may be noisy.
- **Risks**: Medium. Auto-committing in build.sh may surprise users or conflict with CI/CD workflows that expect to handle commits.
- **Best for**: Fully automated pipelines where build.sh owns the commit lifecycle.

## Architecture Notes

- The ArgoCD Application template (`init/argocd/application.tmpl.yaml:11`) hardcodes `path: k8s`. This path must exist and contain valid YAML in the app's git repo for ArgoCD to sync.
- `init.sh` renders templates into the app directory and does an initial git commit (line 282-287). The gitignore prevents these files from being committed.
- `build.sh` re-renders templates at build time (step_template, line 251-333) but does NOT commit them. The user must manually commit after build.
- The existing spec `001-fix-init-templates` suggests k8s/ should NOT be in the scaffolded output (should be generated by build.sh). This conflicts with the ArgoCD requirement that k8s/ must be in the git repo.

## Risk Register

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Removing gitignore rules exposes k8s/ to accidental edits | Low | Medium | Developers already edit these files; git tracks changes for review |
| build.sh auto-commit conflicts with CI/CD workflows | Medium | High | Don't auto-commit in build.sh; keep commit responsibility with user/CI |
| Existing scaffolded apps still have the old gitignore | High | Medium | Users must re-run init.sh or manually fix their .gitignore |

## Open Questions

- Should build.sh also commit the rendered manifests, or leave that to the user/CI?
- Is the spec 001 assumption (k8s/ should NOT be in scaffolded output) still valid, or should it be updated to reflect that k8s/ MUST be in the repo for ArgoCD?
- Are there cases where k8s/ should be gitignored (e.g., when ArgoCD is not used)?

## Recommended Direction

**Approach A** (remove gitignore rules) is the simplest and most correct fix. The k8s/ and argocd/ manifests are required for ArgoCD to function. Gitignoring them creates a contradiction: init.sh renders them, but they're never committed. The gitignore rule was likely added without considering the ArgoCD workflow.

The fix is a two-line change in `init/gitignore`: remove `k8s/*.yaml` and `argocd/*.yaml`. This ensures init.sh's `git add .` captures the rendered manifests, and ArgoCD can find them in the repo.

No changes to build.sh are needed for this fix — build.sh already renders the files into the correct location. Whether build.sh should auto-commit is a separate concern (and likely better handled by CI/CD).
